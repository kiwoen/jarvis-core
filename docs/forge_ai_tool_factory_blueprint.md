> 代号：Forge（熔炉）

## 零、写在前面

这不是又一个"套壳 ChatGPT"的项目。这不是又一个"用别人的 API 搭积木"的方案。

**Forge 的底层逻辑是：让人工智能去创造人工智能工具。**

当一个中小企业主说"我需要一个能自动回复客户微信、同时同步到 ERP 的工具"，Forge 不会回答"对不起，我没有这个功能"。Forge 会自主分析需求、设计架构、编写代码、测试部署，然后在 30 分钟内交付一个**可用的、定制化的、从零编写的** AI 工具。

这就是 Forge 的核心壁垒：**不是"能做什么"，而是"能创造什么"。**

## 一、项目总览

### 1.1 一句话定义

**Forge 是一个需求驱动的自进化 AI 工具自动生成平台——用户用自然语言描述需求，系统自主完成工具的设计、编码、测试、部署和持续进化。**

### 1.2 核心差异化

| 维度 | 现有 AI 产品 | Forge |
|------|-------------|-------|
| **定位** | 提供固定功能的 AI 工具 | 生成任意功能的 AI 工具 |
| **能力边界** | 预定义，不可扩展 | 无边界，需求驱动扩张 |
| **代码来源** | 人工编写 + AI 辅助 | AI 自主从零编写 |
| **知识产权** | 使用第三方 API/框架 | 完全自主生成，可申请专利 |
| **盈利模式** | 订阅费 | 工具生成费 + 市场抽成 + B2G |
| **进化方式** | 版本迭代 | 实时自我进化 |

### 1.3 适用场景矩阵

**多领域 × 多维度 × 多层次 × 多方向**

| 领域 | 维度 | 典型需求 → Forge 自动生成的工具 |
|------|------|-------------------------------|
| **政务服务** | 效率提升 | "我需要一个能自动审核营业执照真伪的插件" → 营业执照智能核验工具 |
| **中小企业** | 成本降低 | "帮我做一个能自动回复客户询价、生成报价单的助手" → 智能客服+报价工具 |

**自研组件清单（不依赖第三方）**：

| 组件 | 功能 | 技术路线 |
|------|------|---------|
| 沙盒引擎 | 安全的代码执行环境 | 进程级隔离 + seccomp + 自研策略引擎 |
| 任务调度器 | 优先级队列 + 资源感知调度 | 自研，Go 实现 |
| 权限管理 | 五级权限 + 审批流 | 自研 RBAC + ABAC 混合模型 |
| 向量记忆总线 | 多级缓存 + 语义检索 | 自研 embedding 管线 |
| 结构化知识图谱 | 实体-关系-事件三元组存储 | 自研图引擎 |
| Forge SDK | 工具开发接口 | 自研 Python/JS SDK |

## 五、技术实现详解

### 5.1 代码基因库（Code Gene Pool）

这是 Forge 最核心的技术，类比生物基因库：

```
基因结构:
  {
    "gene_id": "G20260712_001",
    "type": "api_adapter",           // 基因类型
    "domain": "social_media",         // 领域
    "platform": "douyin",             // 平台
    "function": "comment_fetch",      // 功能
    "language": "python",             // 语言
    "code": "...",                    // 代码片段（已脱敏、参数化）
    "dependencies": ["requests", "playwright"],
    "params": ["account_token", "video_id", "max_count"],
    "quality_score": 0.92,            // 质量分（从实际运行中学习）
    "success_rate": 0.87,             // 成功率
    "usage_count": 234,               // 被引用次数
    "evolved_from": ["G20260601_034", "G20260615_089"],  // 进化来源
    "mutations": ["added_rate_limit", "improved_error_handling"],
    "created_at": "2026-07-12",
    "last_used": "2026-07-12"
  }

基因操作:
  CROSSOVER(gene_a, gene_b):
    → 提取 gene_a 的 接口适配 部分 + gene_b 的 数据处理 部分
    → 融合为新的 api_adapter_with_processing 基因
    
  MUTATE(gene, context):
    → 根据新的平台要求，修改认证方式
    → 添加新的错误处理逻辑
    → 调整参数结构
    
  SELECT(pool, fitness_function):
    → 根据需求匹配度 + 代码质量 + 历史成功率
    → 选出最适合的基因组合
```

### 5.2 需求解析引擎

```
输入示例: 
  "我们公司是做建材的，每天要回复几百条微信消息，
   能不能做一个自动回复的工具，能识别客户问的是价格还是库存，
   然后自动从我们的 Excel 表格里查数据回复？"

解析流程:
  
  Step 1: 领域识别
    → 建材行业 + 客户服务
  
  Step 2: 功能树提取
    ├── 核心功能: 自动回复微信消息
    ├── 子功能1: 意图识别（价格咨询 vs 库存查询）
    ├── 子功能2: 数据查询（读取Excel）
    ├── 子功能3: 回复生成（基于查到的数据）
    └── 子功能4: 消息接入（微信API）
  
  Step 3: 约束提取
    ├── 平台: 微信
    ├── 数据源: Excel表格
    ├── 行业术语: 建材相关
    ├── 回复风格: 商务但不生硬
    └── 预期量级: 数百条/天
  
  Step 4: 已知模式匹配
    → 匹配到"微信客服工具"基因簇（匹配度82%）
    → 匹配到"Excel数据查询"基因（匹配度95%）
    → 匹配到"意图识别"基因（匹配度78%）
  
  Step 5: 缺失识别
    → "建材行业术语理解" → 需要新基因
  
  Step 6: 输出结构化需求规格
    → 传递给架构设计阶段
```

### 5.3 自研沙盒引擎

```
不依赖Docker，从零构建的轻量级沙盒：

架构:
  ┌────────────────────────────┐
  │      沙盒管理器             │
  │  ┌──────┐  ┌──────┐       │
  │  │策略引擎│  │资源限制│      │
  │  │Policy │  │Cgroup │      │
  │  └──────┘  └──────┘       │
  └────────────────────────────┘
            │
  ┌────────────────────────────┐
  │     隔离执行环境             │
  │  ├── 进程隔离 (namespace)  │
  │  ├── 文件系统隔离 (chroot)  │
  │  ├── 网络隔离 (iptables)   │
  │  ├── 系统调用过滤 (seccomp) │
  │  └── 资源限制 (cgroups)    │
  └────────────────────────────┘
            │
  ┌────────────────────────────┐
  │     执行审计                 │
  │  ├── 系统调用日志            │
  │  ├── 文件访问日志            │
  │  ├── 网络请求日志            │
  │  └── 异常行为检测            │
  └────────────────────────────┘

安全等级:
  Level 0: 完全隔离（无网络、只读文件系统）
  Level 1: 只读网络（可DNS+HTTP GET，不能写文件）
  Level 2: 受限写入（沙盒内可写，不能访问外部）
  Level 3: API白名单（只能访问指定的API端点）
  Level 4: 完全访问（需要人工审批）
```
